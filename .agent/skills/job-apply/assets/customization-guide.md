# 履歷客製化策略指引

## 核心原則

1. **誠實呈現** — 只調整呈現方式與重點，不虛構經驗或技能
2. **關鍵字優化** — 使用 JD 中的用詞，幫助通過 ATS（Applicant Tracking System）篩選
3. **相關性排序** — 最相關的經驗和技能放最前面
4. **量化成果** — 盡可能用數字呈現成就（%、人數、時間）

## 各區段調整策略

### Professional Summary

| 情境 | 調整方式 |
|------|----------|
| JD 強調前端 | 摘要開頭提及前端經驗年資，弱化後端描述 |
| JD 強調 AI | 突出 AI 整合經驗，提及具體 AI 工具或框架 |
| JD 強調團隊領導 | 加入帶領團隊、mentoring 的描述 |
| JD 強調全端 | 平衡前後端描述，強調跨領域能力 |

**模式：** `擁有 N 年{JD最相關領域}經驗的{JD職稱}，專精於{JD核心技能}。{最強匹配項的一句話描述}。{次強匹配項}。`

### Core Skills — Skill Bars

- 將 JD 最需要的技能排在第一位
- 百分比可微調（±5%），但不應大幅虛報
- 若 JD 要求的技能不在清單中但確實有經驗，考慮替換最不相關的項目
- 最多顯示 4 項，選擇與 JD 最匹配的組合

### Core Skills — Tech Stack

- 調整分類標籤以呼應 JD 的技術分類方式
- 將 JD 提到的技術排在各分類的前面
- 補充 JD 要求但未列出的技術（必須是真實使用過的）
- 移除與目標職缺完全無關的冷門技術

### Work Experience

| 調整項目 | 具體做法 |
|----------|----------|
| 成就描述 | 用 JD 的關鍵字重新措辭，不改變事實 |
| 重點排序 | 與 JD 最相關的成就放第一個 bullet |
| 補充細節 | 若 JD 要求某技能，在相關經驗中明確提及使用過 |
| 量化數據 | 確保關鍵成就有數字佐證 |

**動詞選擇：**
- 技術類：主導、設計、實作、優化、遷移、整合
- 領導類：帶領、指導、推動、建立
- 成效類：提升、縮短、降低、達成

### Side Projects

- 調整 description 以突出與 JD 相關的技術面向
- 若專案使用了 JD 要求的技術，在 tags 中明確列出
- 考慮調整 category 以更貼近 JD 的領域
- 最相關的專案應排在第一位

### Hero / About

- subtitle 應反映目標職位的核心定位
- typing text 可調整為與 JD 領域相關的句子
- about 的 focus 項目對應 JD 的三個主要需求方向

## ATS 關鍵字策略

1. **直接匹配** — JD 寫 "Vue.js"，履歷也寫 "Vue.js"（不要只寫 "Vue"）
2. **完整名稱** — 同時列出縮寫和全名（如 "CI/CD (Continuous Integration/Deployment)"）
3. **技能區呼應** — JD 的 Required Skills 應在 Core Skills 區段中逐一對應
4. **自然融入** — 關鍵字要融入句子中，不要硬塞成列表
